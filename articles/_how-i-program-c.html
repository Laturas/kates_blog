My Personal C Programming Style

February 17 2026

If there's one thing programmers can't agree on, it's formatting and code styling. The problem is ultimately the discussions are based in personal aesthetic preferences. However, I do think that the argument over code aesthetics is both important and has real interesting things to discuss. I've spent a lot of time in the past thinking about code styling and at this point I believe I've settled on a style that really works for me.

This is in no way an "objectively best way to style your code", rather, it's the style I've settled on as my preferred way to write it after quite a bit of time. I do have arguments for why I think my code style is good, which is what I will be focusing on in this article, but ultimately what matters is what you're productive in. Also it goes without saying, but if you're working in someone else's codebase, you should code in whatever pre-established style there is. Consistent styling is more important than anything.

Goals of Programming Style

I believe you should aim for your style of programming to facilitate your goals with whatever codebase you're in. As such, my goals with code in which my style should facilitate, are the following:

- The code should be accessible. I want my code to be readable by people with limited programming knowledge. We all remember the times way back when we first stepped foot in an established codebase as a new programmer and got lost immediately in hidden knowledge, obscure language features, and bad documentation. I want anyone, including people newer to programming, to be able to learn from the code I write.

- The code should be consistent. Consistency in a codebase helps massively with both onboarding, and being able to navigate it. Consistent conventions maximize productivity for everyone.

- The code should be reliable. Building your code on top of strong foundations which are performant, free of bugs, and have stable API's, will make your code last for decades. People argue for "maintainability" in code, which I believe is the wrong attitude. Maintainable code is not the goal. The best code is the code that works reliably, is clearly documented and readable, and never has to be touched for decades. You don't want a car that is easy to fix but breaks constantly, you want a car that never breaks.

- The code should be navigable. It should be as trivial as possible to jump to any arbitrary block of code you want. So it should be organized in a logical manner in a way that it's easy to deduce where to find a specific piece of code. As well, in my opinion, the king of navigation is find (ctrl+f). Nearly every text editor in existence has a find feature. Thus I use an even stricter requirement than "it should be greppable". I don't want to type a regex to go to a function or struct definition. I want to just search for it.

C Standard Version

I strictly use a subset of C99. I don't use the standard library at all where I can, I don't use VLA's, and I don't use features that are now deprecated, optional, or broken in later C standards. This is because I want my code to be easily portable to other platforms and C99 is the second most widely supported C standard.

Ok yeah, C89 is more portable, but this isn't the early 2000's anymore. Especially since Microsoft finally stopped dragging their feet, C99 is well supported on dang near any platform in existence today, and will have to be well supported on future platforms or such a platform will miss out on many of the modern C projects that keep computing alive.

If we look in practical terms, a quick glance at the C99 implementations page on Wikipedia shows that a reasonable subset of C99 is supported by nearly any C compiler in existence today.

Naming Things

I have a strict set of rules for naming things in my code base. For actual formatting of the names:

function_names() TypeNames variable_names MACRO_IDENTIFIERS ENUM_IDENTIFIERS

I have settled on this for a while, and believe I have decent arguments for it beyond just aesthetics (though, I suppose that's what all of this is at the end of the day). I want my code to be readable with no syntax highlighting, so there are important things to distinguish. Macros need to be clearly identifiable so they are in UPPER_SNAKE_CASE. I choose to make enums named the same way, because at the end of the day there's little difference between a #define'd constant and an enum member in terms of functionality for the code.

Type names are the most important to distinguish after that. PascalCase is ideal for this purpose, because type names should be really short (in my opinion, they should not exceed 3 words), are typed often, and should convey importance. PascalCase doesn't add extra characters, is readable when short, and is easy to type. I believe the uppercase letters in PascalCase also somewhat signal importance, as they're large and strong.

Lastly, variable and function names both use lower_snake_case. You may argue that this adds confusion, but functions rarely appear without parentheses immediately following them. The only time this happens is with function pointers - and if that happens I clearly delineate that in the name of any such functions via postfixing "fptr". Function pointers are very rare in my codebase anyways though, as I find they often add more complexity than they take away.

I believe lower_snake_case is the easiest case to read as it is closest to how we read text. Being able to read code is more important than being able to write it, thus variables and functions (being the two most common things in a codebase) should have maximum readability.

This makes variable and function names quite long. Yes, this makes the code slightly slower to type, but the amount of time you spend typing is miniscule in terms of the actual process of software development - especially if you work with a text editor with a working autocomplete. Typing code rarely matters - reading code always matters. In the tradeoff between typing speed (writeability) and clarity (readability), clarity should always win.

Typedefs

I always typedef structs and enums immediately. Having to constantly type struct or enum before the type in my opinion is both a waste of time and makes the code harder to read. It adds unnecessary noise, rarely adds information that can't be easily inferred, and doesn't solve issues with namespace collisions.

Another massive benefit this adds is that I can trivially jump to the definition by simply searching for "struct name" or "enum name". This will always take me to the definition because this is the only place where the struct or enum keywords appear. This comes up in practice more than you'd think.

I also typedef my own basic numeric types

u8, u16, u32, u64 - Unsigned integers

i16, i32, i64 - Signed integers

f32, f64 - Floating point numbers

I typedef these because they're used extremely often, and the default C naming conventions are terrible. I've seen some people claim this is unreadable but:

A - you can't tell me "unsigned long long" is more readable to the average person than "u64"
B - These naming conventions don't come from nowhere. They're already used in languages like Rust, Odin, and Zig.
C - These names provide valuable information you don't glean from the standard C names. "int" could theoretically be 16 bits. "long int" is 32bits on Windows and 64bits on Linux. But an i64 is only one possible thing - a signed 64 bit integer. I can typedef whatever weird platform requirements I need to make it fit.

I've also seen the claim that typedef'ing these makes it more likely for namespace collisions if I integrate someone else's codebase, but in practicality what that means is that I delete my typedefs and use theirs instead. The beauty of the names is that they clearly delineate their properties so I can be sure they won't cause issues.

Pointers

I never typedef pointers. Anyone that does this needs to be tried for war crimes. The only case it would be useful is avoiding a problem I'll discuss in just a second.

While we're on the topic of pointers, I also put the * with the pointer type. My argument for this is that the * indicates it's a pointer type, so it should... go with the type. I suspect the reason that this isn't done often is that you run into a problem with this code:

int* a, b, c;

In the above code, only a is a pointer. b and c are both just integers because the * only applied to a. This is also why I would guess people typedef pointers. There's an easy way around this though. Don't write code like the above! Each variable declaration gets its own line. Which, yes, inflates the line count, but in my opinion it's clearer and it encourages you to declare variables where they're used or give them an initialization.

int* a = NULL; int** b = &a; int* c = some_function();

This is also what people coming from other languages are more likely to expect. It's more readable and less error prone for everyone.

No Namespaces?

This is one of the biggest annoyances people cite with C. However, I've found it's not too hard to work around this with good naming conventions.

I prefix all enum members with the same prefix.

typedef enum EntityID { EID_NONE = 0, EID_PLAYER, EID_MONSTER, EID_BOSS, /* ... */ } EntityID;

And I name functions in a very specific way. It is always "module_data_action". I feel very strongly that this is the best way to name functions. For example, let's consider some functions from the module I made to draw the WIP editor UI in my game engine. As a challenge, see if you can determine what they do from their function signatures!

bool ui_region_is_mouse_hovered(Rectangle region); MouseState ui_element_mouse_state(Rectangle element_rect); void ui_editor_object_list_draw(i32 x_pos, i32 y_pos); void ui_editor_draw();

The names themselves begin to act like namespaces! I immediately know where in my code a function named ui_... is located, and then what it acts on and what it does. There is much debate over long names, but I fall squarely into the camp of "descriptive names are good". I want someone reading my code, with no prior knowledge about the internals, to understand what is going on and where to find things.

Fun with Whitespace

Ah yes, the great debate over spaces vs tabs. I personally think this debate is nuts because as far as I'm concerned there's a correct answer here. Tabs all the way.

Argument 1: Tabs are consistent with scope rules. The rule is obvious and impossible to break. 1 tab = 1 scope level. That's it. Indentation is meant to indicate scope, so if you're using say 4 spaces, the idea you could even have such a thing as half indentation is a mistake. Similarly,

Argument 2: Tabs are easier to work with. The fact that 1 tab = 1 character = 1 scope is a strict rule means that it's trivial to work with them no matter your editor. If your editor doesn't have auto indentation or scope lines or something along those lines it's way easier to deal with tabs than spaces in my opinion.

Argument 3: Tabs cost less. This doesn't matter as much I suppose but I figure I'll throw it in anyways. A tab character is 1 byte. 4 spaces is 4 bytes and 8 spaces is 8 bytes. Does this matter? The Linux kernel uses tabs instead of spaces and an indentation width of 8. If we assume each line has an average of 1 scope, and the codebase is ~35 million lines, this translates into a difference of 35Mb to 280Mb of file size out of the roughly 1.7GB of code (excluding version control, though that wouldn't change the percentage most likely). That's nothing to sneeze at, I feel!

You may argue that the fact that the tab character doesn't have a consistent size would be bad. Alright, cool, but I don't know how much this actually matters in practice? If you have column width requirements, just specify how wide a tab is and what code will be judged by. If you absolutely need your code to appear exactly the same completely unaltered between VSCode and the GitHub code viewer, go off I guess?

Also, I don't use tabs for aligning things horizontally between lines. Spaces are indeed better for that. But that's a distinctly different use case than what I am arguing.

Newlines

I use the K&R style with the curly brace on the same line as the item declaring the scope

if (condition) { /* Code */ }

This is so that the start of a scope and end of a scope are the same - they're 1 line. It makes it clearer in my opinion the relation between the opening scope and the thing opening it. However, I must state I don't feel that strongly about this. The Allman style of having the curly brace on the next line is fine and readable, just not what I personally write.

Any other formatting style of braces and newlines are horrible though (looking at you GNU). Please use K&R or Allman style, I beg of you.

Similarly, I try to keep short function declarations all on one line. If it's long, then I break the arguments onto their own lines. I don't have a hard rule for when a function definition is "too long". It's mostly vibes based. Extra note: typedef-ing structs and enums also helps keep function definitions on one line which is another reason I do that.

/** * Short, single line declaration */ void short_function(int arg1, int arg2) { /* Code */ } /** * Many argument definition */ void long_function( int arg1, int arg2, const char* arg3, int* arg4, ) { /* Code */ }

I do the same for when I am calling functions, but I don't need to break those as often because the lack of types reduces the length of the lines significantly.

To Space or Not to Space?

I put spaces around math operations

int y = (x * 5) + 3

I also tend to parenthesize quite a lot, even when not necessary for maximum clarity (like the above). Even if I removed the parentheses because of operator precedence it would parse correctly, but I think the parentheses help make it more immediately clear when skimming.

I put spaces between ifs/whiles/fors/etc and their arguments, but I don't for functions and macros.

/* Must be 16 bytes for alignment reasons */ ASSERT(sizeof(I32Array) == 16) I32Array array = config_file_read(); bool is_valid = validate_array(array); if (is_valid) { for (int i = 0; i < I32Array.length; i++) { printf("%d\n", array[i]); } }

Also, ifs/fors/whiles/etc always need a curly brace following them, even if it's all on the same line. If it is on the same line I like to add spaces for a bit more breathing room. It's always a pain in the rear to fix it otherwise if you need to add more functionality.

/* ok */ if (condition) { function(); } /* not ok */ if (condition) function();

Macros

Don't use them for anything complicated. I only use macros for a few operations:

- #define-ing constants that don't work inside enums.

- Extremely simple functions, like an absolute value, so I can guarantee they get inlined

- Getting compile time information with things like sizeof, __LINE__, __FILE__, etc.

- Assertions and related

- Replacing existing function calls in your codebase for debug purposes (e.g. #define malloc(x) debug_malloc(x, __LINE__, __FILE__))

That's basically it. Macros are dangerous because the error messages from them can be horrifying and they can be hard to debug. For example, I think the Linux kernel's embedded data structure macros are awful to use. Just embed and use data structures manually, or if you really have to, use a function so I get type checking with errors that are actually readable.

Build system

You may argue this extends a little bit out of the code styling territory. However, it influences how you organize your code so I think it's fine to include here.

What build system do I use? I don't use one. At all. Make and its consequences have been a disaster for humanity.

I do what the industry calls a "unity build". The entirety of my code is a single translation unit that gets chewed through in a straight line.

I have a single header file that is included by the main.c file, that itself includes every file in the project that needs to be built in its dependency order.

For the game engine I am working on, thus, this is the entire build script for the project:

gcc main.c -o afterhours.exe \ -g3 -O0 \ -I libs/include \ libs/lib/windows/libraylib.a \ -lm -lpthread -ldl -lgdi32 -lwinmm

Having a complete lack of a build system has a number of advantages. For one, absolutely any basic C lsp can understand the codebase immediately. You don't have to explain to it the build order or anything. This matters way more than you think - because I've had to constantly fiddle with Vscode's LSP for various projects I've interacted with because everything is abstracted away by the build system. It's awful and sucks.

Another advantage is build time. You'd think that unity builds would be slower than incremental, but I rarely find this to matter in practice. For one, incremental builds bring with them overhead of their own. Their advantages only begin to appear with truly massive projects. I currently just use gcc since it's what I have on my system and is fast enough currently, but considering during development I don't want heavy optimizations, I could always switch to compilers that actually compile C in a reasonable amount of time. As an example, TCC can compile tens of thousands to a hundred thousand lines of code a second.

There is no world where my game + engine exceeds a few hundred thousand lines of code.
